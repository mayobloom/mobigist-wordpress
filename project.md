# WordPress Personal Knowledge Blog 개발 명세서

## 1. 프로젝트 개요

### 1.1 목적

WordPress 기반의 개인 블로그를 구축한다. 기본 블로그 기능에 더해 회원 권한 관리, WordPress 블록 에디터 기반 작성 경험, 수식 렌더링, WordPress 내부 링크, Knowledge Graph를 제공하여 글과 글 사이의 지식 연결을 시각화한다.

### 1.2 구축 방식

- 로컬 환경에서 Docker Compose 기반으로 개발 및 테스트한다.
- 운영 환경은 AWS Lightsail 배포를 전제로 한다.
- 도메인은 AWS Route 53에서 관리하는 도메인을 사용한다.
- 메일 발송은 AWS SES 연동을 우선 검토한다.
- WordPress 기본 기능과 검증된 플러그인을 우선 활용한다.
- 기본 기능 또는 플러그인으로 구현이 어렵거나 유지보수성이 낮은 기능만 커스텀 플러그인으로 개발한다.

### 1.3 우선순위

모든 기능은 최종적으로 구현한다. 단, 개발 순서는 다음과 같이 나눈다.

- 1차 구현: 블로그 운영에 필요한 기반 기능
- 2차 구현: 작성 경험, 검색, 권한, 운영 편의성 고도화
- 3차 구현: WordPress 내부 링크와 Knowledge Graph
- 4차 구현: 성능, 보안, 운영 자동화, 배포 안정화

## 2. 기술 스택 및 운영 환경

### 2.1 기본 스택

- CMS: WordPress
- 테마: Shhh
- DB: MariaDB 또는 MySQL
- 실행 환경: Docker, Docker Compose
- 웹 서버: WordPress 공식 Docker 이미지 기본 구성을 우선 사용하고, 운영 배포 시 Lightsail 환경에 맞게 조정한다.
- 버전 관리: Git 사용을 전제로 한다.

### 2.2 Docker 구성

필수 서비스는 다음과 같다.

- `wordpress`: WordPress 애플리케이션 컨테이너
- `db`: MariaDB 또는 MySQL 컨테이너
- `phpmyadmin` 또는 동등한 DB 관리 도구: 로컬 개발 환경에서만 사용
- 선택 서비스: Redis, Mailpit 또는 Mailhog, WP-CLI

### 2.3 환경변수 관리

환경변수는 `.env` 파일로 관리한다. 예시는 `.env.example`에 제공한다.

필수 항목:

- WordPress DB 접속 정보
- WordPress 관리자 초기 계정 정보
- 사이트 URL
- AWS SES SMTP 정보
- 운영/개발 환경 구분값
- 보안 키 및 salt 값

민감 정보가 포함된 `.env` 파일은 Git에 포함하지 않는다.

## 3. 디자인 명세

### 3.1 테마

- Shhh 테마를 사용한다.
- 테마 원본은 직접 수정하지 않고, 필요한 경우 child theme 또는 커스텀 플러그인에서 스타일을 확장한다.
- 추가 UI는 Shhh 테마의 미니멀한 글쓰기 중심 디자인과 시각적으로 일관되게 구현한다.

### 3.2 폰트

- 기본 폰트는 Nanum Gothic Coding을 사용한다.
- 폰트 파일은 운영 서버에서 로컬 호스팅한다.
- 외부 Google Fonts 호출은 사용하지 않는다.

### 3.3 디자인 방향

- 키워드: 미니멀리즘, 간결함, 편안함, 필사 노트
- 글 본문 가독성을 최우선으로 한다.
- 과한 장식 요소, 큰 배너, 마케팅형 랜딩 페이지 구성은 사용하지 않는다.
- 썸네일은 글 목록에서 작음에서 중간 크기의 미리보기 이미지로 노출한다.

## 4. 기능 범위

### 4.1 1차 구현: 기반 기능

#### 4.1.1 WordPress 설치 및 기본 설정

- Docker Compose로 WordPress와 DB를 실행할 수 있어야 한다.
- 로컬에서 초기 설치, 테마 적용, 기본 플러그인 설치가 가능해야 한다.
- 관리자 계정으로 로그인하여 기본 사이트 설정을 완료할 수 있어야 한다.

#### 4.1.2 게시판 구조

WordPress의 카테고리 계층 기능을 사용하여 게시판을 구현한다.

대분류 카테고리:

- News: 뉴스레터
- Review: 책, 영화, 음악 등 다양한 작품에 대한 감상평
- Academic: 논문, 브리핑, 월간지, 학습 내용과 통찰
- Note: 일기, 성찰, 자기반성
- About: 블로그 소개
- Admin: 관리자 및 운영자 전용 영역

요구사항:

- 카테고리는 최대 3단계까지 허용한다.
- 관리자는 카테고리의 생성, 조회, 수정, 삭제를 할 수 있어야 한다.
- 홈페이지 상단 또는 주요 내비게이션에는 대분류 카테고리명과 게시글 수를 `카테고리명 (게시글 수)` 형식으로 표시한다.
- Admin 카테고리 또는 Admin 전용 페이지는 권한이 없는 사용자에게 노출하지 않는다.

#### 4.1.3 포스팅 기본 기능

WordPress 기본 글 기능을 우선 사용한다.

요구사항:

- 관리자는 게시글을 생성, 조회, 수정, 삭제할 수 있다.
- 일반 방문자와 회원은 공개 게시글을 조회할 수 있다.
- 게시글에는 제목, 본문, 카테고리, 태그, 대표 이미지, 발행 상태, 공개 범위, 작성일, 수정일을 설정할 수 있어야 한다.
- 대표 이미지는 글 목록 화면에 썸네일로 표시한다.
- `#태그명` 입력을 통한 자동 태그 생성은 2차 구현에서 검토하되, 1차에서는 WordPress 기본 태그 기능을 사용한다.

#### 4.1.4 댓글 기본 기능

WordPress 기본 댓글 기능을 사용한다.

요구사항:

- 회원은 댓글을 작성할 수 있다.
- 회원은 본인이 작성한 댓글을 수정 또는 삭제할 수 있어야 한다.
- 회원은 댓글에 대댓글을 작성할 수 있어야 한다.
- 대댓글은 WordPress threaded comments 기능을 우선 사용한다.
- 관리자는 모든 댓글을 관리할 수 있다.
- 비회원 댓글은 허용하지 않는다.
- 댓글 승인, 스팸 방지, 신고 또는 제재 기능은 2차 구현에서 확장한다.

#### 4.1.5 REST API 기본 기능

WordPress REST API를 사용한다.

요구사항:

- 게시글 목록 조회, 단일 게시글 조회, 댓글 조회는 공개 API로 제공한다.
- 게시글 생성, 수정, 삭제는 관리자 권한 인증 후 가능해야 한다.
- 댓글 생성, 수정, 삭제는 사용자 권한에 따라 제한한다.
- WordPress 기본 REST API 엔드포인트를 우선 사용한다.
- 커스텀 엔드포인트는 좋아요, 내부 링크, Knowledge Graph 등 기본 API로 표현하기 어려운 기능에 한해 추가한다.

### 4.2 2차 구현: 회원, 검색, 작성 경험 고도화

#### 4.2.1 회원가입 및 마이페이지

요구사항:

- 사용자는 회원가입, 로그인, 로그아웃을 할 수 있다.
- 회원가입 직후 이메일 인증을 완료해야 계정이 활성화되어야 한다.
- 인증 전 사용자는 `pending_email_verification` 상태로 관리한다.
- 인증 전 사용자는 로그인, 댓글 작성, 대댓글 작성, 게시글 좋아요, 댓글 좋아요, 마이페이지, 특별회원 기능을 사용할 수 없다.
- 인증 링크에는 만료 시간이 있는 토큰을 사용한다.
- 인증 메일 재발송 기능을 제공한다.
- 인증되지 않은 계정은 지정 기간이 지난 뒤 자동 삭제할 수 있어야 한다.
- 회원은 마이페이지에서 기본 프로필을 확인하고 수정할 수 있다.
- 회원은 본인이 작성한 댓글과 좋아요한 글 목록을 확인할 수 있다.
- 회원은 회원탈퇴를 요청할 수 있다.
- 회원탈퇴 시 계정, 작성 댓글, 대댓글, 게시글 좋아요, 댓글 좋아요 데이터 처리 정책을 명확히 적용한다.

UI/UX 기준:

- 회원가입, 로그인, 비밀번호 재설정, 이메일 인증 안내, 인증 메일 재발송, 마이페이지, 회원탈퇴 화면은 Shhh 테마와 시각적으로 조응하는 프론트엔드 화면으로 제공한다.
- WordPress 기본 `wp-login.php` 화면은 일반 사용자에게 직접 노출하지 않는 방향으로 구현한다.
- 회원 관련 화면은 미니멀하고 글 중심적인 Shhh 디자인을 해치지 않도록 간결한 폼, 충분한 여백, 낮은 대비의 보조 텍스트를 사용한다.
- 플러그인을 사용하는 경우에도 기본 플러그인 UI를 그대로 노출하지 않고, 가능한 한 커스텀 템플릿, shortcode, block, CSS override로 사이트 디자인에 맞춘다.

구현 방식:

- WordPress 기본 회원가입 기능은 회원가입과 알림 메일 발송은 지원하지만, 가입 직후 이메일 인증을 통한 계정 활성화 플로우는 부족하므로 단독 사용하지 않는다.
- 이메일 인증, 프론트엔드 회원가입 UI, 로그인 UI, 마이페이지를 제공하는 회원관리 플러그인을 우선 검토한다.
- 플러그인이 인증 상태, 권한 제한, 회원탈퇴 즉시 삭제, Shhh 디자인 요구사항을 충분히 만족하지 못하면 커스텀 플러그인으로 보완한다.
- 사용자 데이터 모델과 role/capability는 WordPress 기본 사용자 시스템을 사용한다.
- 인증 상태, 인증 토큰, 토큰 만료 시간, 인증 메일 재발송 이력은 user meta 또는 커스텀 테이블로 관리한다.
- 인증 메일 발송은 AWS SES SMTP 연동을 우선 사용한다.

회원탈퇴 기본 정책:

- 회원탈퇴가 확정되면 회원 계정과 회원이 작성한 댓글, 대댓글, 게시글 좋아요, 댓글 좋아요 데이터를 완전히 삭제한다.
- 탈퇴 요청이 접수되면 철회 가능 유예기간 없이 즉시 삭제한다.
- 법령상 보존 필요성이 있는 데이터가 생기는 경우에는 해당 데이터만 별도 보관 영역에 분리 보관한다.
- 법정 보존 대상 데이터는 서비스 데이터베이스의 일반 조회, 검색, 그래프, 통계에서 제외한다.
- 법정 보존 기간이 만료되면 별도 보관 데이터도 완전히 삭제한다.
- 현재 프로젝트가 단순 블로그로 운영되고 유료 결제, 재화 공급, 소비자 분쟁 처리가 없다면 회원탈퇴 시 즉시 완전 삭제를 기본 정책으로 한다.
- 향후 유료 구독, 후원, 결제, 상품 판매 기능이 추가될 경우 전자상거래법상 거래기록 보존 의무를 별도로 검토한다.

#### 4.2.2 권한 체계

WordPress roles and capabilities를 기반으로 구현한다.

권한 그룹:

- 관리자
- 매니저
- 특별회원
- 일반회원

권한 요구사항:

- 관리자는 시스템의 모든 권한을 가진다.
- 관리자는 게시글, 댓글, 카테고리, 회원, 회원 권한을 생성, 조회, 수정, 삭제할 수 있다.
- 매니저는 특별회원과 일반회원의 댓글을 관리할 수 있다.
- 매니저는 회원 제재 기능을 사용할 수 있다.
- 특별회원은 일반회원 권한에 더해 특별회원 전용 게시글을 조회하고 좋아요 및 본인 댓글 관리를 할 수 있다.
- 일반회원은 공개 게시글 조회, 좋아요, 본인 댓글 생성, 수정, 삭제를 할 수 있다.
- 특별회원 전용 게시글은 카테고리가 아니라 게시글 메타데이터의 접근 권한 값으로 구분한다.

회원 제재 기본 정책:

- 제재 유형은 댓글 작성 제한, 로그인 제한, 계정 비활성화로 구분한다.
- 제재 기간은 영구 또는 지정 기간으로 설정할 수 있어야 한다.
- 제재 사유와 처리자를 기록한다.

#### 4.2.3 좋아요 기능

요구사항:

- 로그인한 회원은 게시글에 좋아요를 누르거나 취소할 수 있다.
- 회원 1명은 게시글 1개에 좋아요를 1회만 등록할 수 있다.
- 게시글 목록과 상세 화면에 좋아요 수를 표시한다.
- 마이페이지에서 좋아요한 글 목록을 확인할 수 있다.
- 로그인한 회원은 댓글과 대댓글에 좋아요를 누르거나 취소할 수 있다.
- 댓글 좋아요 종류는 하트 1종만 제공한다.
- 회원 1명은 댓글 또는 대댓글 1개에 좋아요를 1회만 등록할 수 있다.
- 댓글과 대댓글에는 좋아요 수를 표시한다.
- REST API 또는 Ajax 기반으로 페이지 새로고침 없이 동작하도록 구현한다.

구현 방식:

- 플러그인 검토 후 적절한 플러그인이 없거나 확장성이 부족하면 커스텀 플러그인으로 구현한다.
- 게시글 좋아요 데이터는 사용자 ID와 게시글 ID의 유일 관계를 보장해야 한다.
- 댓글 좋아요 데이터는 사용자 ID와 댓글 ID의 유일 관계를 보장해야 한다.

#### 4.2.4 검색

요구사항:

- 모든 사용자와 비회원 방문자가 검색을 사용할 수 있어야 한다.
- 키워드 검색과 태그 기반 검색을 지원한다.
- 검색 결과는 글 목록과 같은 디자인 원칙을 따른다.
- 검색 결과에는 제목, 요약, 카테고리, 태그, 썸네일, 발행일을 표시한다.
- 검색 결과 정렬을 지원한다.

정렬 기준:

- 최신순
- 오래된순
- 제목순
- 좋아요순
- 댓글 많은 순

구현 방식:

- 1차적으로 WordPress 기본 검색과 태그 아카이브를 활용한다.
- 정확도, 한국어 검색 품질, 정렬 확장이 부족하면 검색 플러그인 또는 커스텀 쿼리를 적용한다.

#### 4.2.5 작성 경험

요구사항:

- 게시글 작성은 WordPress block editor를 기본 작성 환경으로 사용한다.
- WordPress block editor가 제공하는 기본 Markdown 단축 입력, 블록 변환, `/` 블록 삽입 기능을 우선 활용한다.
- 별도 Markdown 에디터 또는 Notion식 에디터 복제는 구현하지 않는다.
- 커스텀 작성 도구는 WordPress block editor와 충돌하지 않는 add-on 방식으로만 제공한다.

구현 기준:

- WordPress block editor의 기본 작성 경험이 충분하므로 이를 기준 구현으로 본다.
- 수식 입력 등 특정 작성 보조 기능은 독립 플러그인 또는 editor extension으로 확장한다.
- WordPress core editor 파일은 수정하지 않는다.

#### 4.2.6 수식 지원

요구사항:

- 게시글 본문에서 LaTeX 문법을 사용할 수 있어야 한다.
- 인라인 수식과 블록 수식을 모두 지원한다.
- 작성 중 수식 미리보기 또는 live math 기능을 제공한다.
- 프론트엔드에서도 수식이 안정적으로 렌더링되어야 한다.

구현 방식:

- MathJax, KaTeX 또는 WordPress 수식 플러그인을 우선 검토한다.
- logseq-live-math는 참고 구현으로 활용하되, WordPress와의 호환성을 별도 검토한다.

#### 4.2.7 해시태그 자동 생성

요구사항:

- 게시글 작성 중 `#태그명` 형식의 텍스트를 태그 후보로 인식한다.
- 저장 시 태그 taxonomy와 연결할 수 있어야 한다.
- 코드블록, URL, 기존 HTML 속성 등에서 오탐지하지 않아야 한다.

구현 방식:

- 기본 태그 기능을 유지한다.
- 자동 생성은 editor extension 또는 저장 시 content parser로 구현한다.

### 4.3 3차 구현: 내부 링크 및 Knowledge Graph

#### 4.3.1 WordPress 내부 링크

요구사항:

- 게시글 본문에서 WordPress block editor의 기본 링크 기능을 사용하여 기존 게시글에 대한 내부 링크를 만들 수 있어야 한다.
- 특정 heading으로 연결하는 경우 WordPress 링크 URL의 fragment, 예: `/post-slug/#heading-slug`, 를 사용한다.
- 링크 생성 UI와 게시글 검색 자동완성은 WordPress block editor의 기본 링크 검색 기능을 우선 사용한다.
- 별도 `[[...]]` 문법과 `[[글 제목#heading]]` 전용 자동완성 UI는 구현하지 않는다.
- 저장 시 본문에 생성된 내부 `<a href="...">` 링크를 파싱하여 내부 링크 관계를 커스텀 테이블에 저장한다.
- 내부 링크 관계는 URL을 `url_to_postid()` 등 WordPress API로 해석하여 가능한 한 게시글 ID 기반으로 관리한다.
- heading 링크는 URL fragment를 `target_heading_slug`로 저장한다.
- 프론트엔드 본문 원본은 WordPress가 저장한 링크 HTML을 그대로 유지한다.

구현 기준:

- 화면 표시용 링크 HTML과 그래프 데이터 추출을 분리한다.
- 내부 링크 인덱싱은 게시글 저장 시점에 수행하여 조회 시 부담을 줄인다.
- 게시글 본문 heading에는 프론트엔드 렌더링 시 안정적인 앵커 ID를 부여한다.
- heading fragment로 이동한 직후 해당 heading 위치를 짧은 하이라이트 애니메이션으로 표시한다.
- 외부 링크는 Knowledge Graph 관계로 저장하지 않는다.
- 같은 사이트의 게시글 링크만 내부 링크 관계로 저장한다.

권장 데이터 구조:

- source post ID
- target post ID
- target heading slug
- original link text 또는 anchor text
- created at
- updated at

#### 4.3.2 Knowledge Graph

요구사항:

- 내부 링크 관계를 기반으로 게시글 간 연결 그래프를 표시한다.
- Obsidian Graph View의 주요 요소를 참고한다.
- 노드는 게시글을 의미한다.
- 엣지는 WordPress 내부 링크 HTML에서 추출한 게시글 간 링크 관계를 의미한다.
- 노드를 클릭하면 해당 게시글로 이동하거나 미리보기 정보를 표시한다.
- 그래프 노드 텍스트를 클릭해도 해당 게시글로 이동해야 한다.
- 카테고리, 태그, 공개 범위에 따라 그래프 필터링이 가능해야 한다.

MVP 범위:

- 메인 페이지의 블로그 설명 섹션 아래에 한 줄 여백을 두고 `Graph View` 텍스트 링크를 표시
- `Graph View` 텍스트를 클릭하면 전체 그래프 조회 페이지로 이동
- 전체 그래프 조회 페이지에 전체 게시글 그래프 표시
- 각 게시글 하단에 현재 글 기준 부분 그래프 표시
- 부분 그래프는 열람 중인 글과 해당 글에서 N단계까지 연결된 글을 표시
- 부분 그래프의 연결 단계 수 N은 관리자 페이지에서 설정
- 부분 그래프의 단계별 opacity는 관리자 페이지에서 설정
- 노드 클릭 시 게시글 이동
- 노드 텍스트 클릭 시 게시글 이동
- 공개 게시글만 비회원에게 노출

확장 범위:

- 카테고리별 필터
- 태그별 필터
- 검색어 기반 필터
- 전체 그래프와 부분 그래프의 연결 깊이 설정
- 고립 노드 표시 여부 설정
- 그래프 레이아웃 설정

관리자 설정 항목:

- 부분 그래프 기본 연결 단계 수
- 단계별 노드 opacity
- 단계별 엣지 opacity
- 고립 노드 표시 여부
- 그래프 표시 위치
- 비회원에게 그래프를 노출할지 여부

구현 방식:

- 기존 WordPress Knowledge Graph 관련 플러그인을 조사한다.
- 적절한 플러그인이 없으면 커스텀 플러그인으로 REST API와 프론트엔드 그래프 UI를 구현한다.
- 그래프 렌더링은 검증된 JavaScript 시각화 라이브러리를 사용한다.

### 4.4 4차 구현: 최적화, 보안, 배포

#### 4.4.1 성능 최적화

요구사항:

## 10. 확정 정책 및 추가 검토 사항

### 10.1 확정 정책

- 회원탈퇴 시 회원 계정, 작성 댓글, 대댓글, 게시글 좋아요, 댓글 좋아요는 완전히 삭제한다.
- 법령상 보존 의무가 있는 데이터가 생기는 경우에는 해당 데이터만 별도 보관하고, 보존 기간 만료 후 완전히 삭제한다.
- 비회원 댓글은 허용하지 않는다.
- 특별회원 전용 게시글은 게시글 메타데이터의 접근 권한 값으로 구분한다.
- 내부 링크는 WordPress 기본 링크 HTML을 원본으로 사용하며, 별도 `[[...]]` 문법은 사용하지 않는다.
- Knowledge Graph는 메인 페이지의 `Graph View` 텍스트 링크, 전체 그래프 조회 페이지, 각 게시글 하단의 부분 그래프를 모두 제공한다.
- 메인 페이지에서는 블로그 설명 섹션 아래에 한 줄 여백을 두고 `Graph View` 텍스트 링크를 표시한다.
- `Graph View` 텍스트를 클릭하면 전체 그래프 조회 페이지로 이동한다.
- 게시글 하단 부분 그래프의 연결 단계 수와 단계별 opacity는 관리자 페이지에서 설정할 수 있어야 한다.
- 폰트는 로컬 호스팅한다.

### 10.2 법적 검토 필요 항목

- 현재 프로젝트가 단순 블로그로 운영되는 경우 회원탈퇴 시 즉시 완전 삭제를 기본 정책으로 한다.
- 향후 유료 구독, 후원, 결제, 상품 판매, 소비자 분쟁 처리 기능이 추가되면 전자상거래법상 거래기록 보존 대상과 기간을 재검토한다.
- 전자상거래법상 거래기록 보존 대상이 발생하는 경우, 개인정보 이용 동의 철회 또는 회원탈퇴 후에도 해당 거래기록과 관련 개인정보를 별도 보관할 수 있으며 일반 회원 데이터와 분리해야 한다.
- 보존 기간 예시는 표시ㆍ광고 기록 6개월, 계약 또는 청약철회 기록 5년, 대금결제 및 재화 공급 기록 5년, 소비자 불만 또는 분쟁처리 기록 3년이다.
- 참고 법령: 개인정보 보호법 제21조 개인정보의 파기, 전자상거래 등에서의 소비자보호에 관한 법률 제6조 거래기록의 보존, 같은 법 시행령 제6조 거래기록의 대상ㆍ범위ㆍ기간.
- 법령 출처: 국가법령정보센터 `https://www.law.go.kr`

### 10.3 현재 미확정 항목

현재 추가 확인이 필요한 정책은 없다.
